# Fractal Ontology

This folder contains notes and artifacts related to **fractal ontology**.

## What fractal ontology means

If taxonomy helps classify what something **is**, ontology helps explain what that thing **means in relation to other things**.

A fractal ontology is a way of structuring those relationships so that the **same relational logic can recur across levels** rather than only working at one scale.

In a normal ontology, you might map entities and relationships in a single domain or at a single layer of abstraction. That can be useful, but it often becomes difficult to maintain when systems become more complex and when the same concepts need to be interpreted across enterprise, team, workflow, document, issue, and task levels.

A fractal ontology is meant to solve that problem.

It assumes that although the object may change in size or context, the same kinds of relational questions can still recur:

- What is this connected to?
- What informs it?
- What depends on it?
- What does it constrain?
- What does it enable?
- Who owns it?
- What evidence supports it?
- What actions can follow from it?

The scale changes, but the relational logic remains legible.

## Why this matters

AI systems do not fail only because they produce wrong answers.

They also fail because they lose context.

That loss of context can take several forms:

- disconnected entities
- missing dependencies
- shallow classification without meaning
- broken inheritance across layers
- inability to see local objects in broader system context
- inconsistent interpretation from one layer to another

A fractal ontology matters because it helps preserve **semantic continuity** across levels.

It gives the system a way to represent not only isolated objects, but the network of meanings and dependencies that make those objects interpretable.

## Core idea

The core claim of fractal ontology is:

**the same kinds of relationship logic should be able to recur across levels of the system.**

That does not mean every object has identical relationships.  
It means each object can be interpreted through a familiar relational frame.

For example, many nodes may still be understandable through recurring relationship types such as:

- parent / child
- informs / is informed by
- constrains / is constrained by
- enables / is enabled by
- depends on / supports
- owned by
- governed by
- evidenced by
- acted on by
- related to

This makes meaning more stable and more portable across contexts.

## Fractal ontology vs. ordinary ontology

### Ordinary ontology
Ordinary ontology is often:
- domain-specific
- difficult to extend cleanly across scales
- rich in meaning but weak in recursive portability
- useful in one layer, but harder to maintain across many

### Fractal ontology
Fractal ontology is:
- recursive
- cross-level
- designed for semantic continuity
- better suited to layered systems
- more useful when humans and AI both need to interpret the same structure

## Relationship to taxonomy

Taxonomy and ontology work together, but they are not the same.

Taxonomy helps answer:
- What is this?
- What category does it belong to?
- What kind of thing is it?
- What level is it at?

Ontology helps answer:
- What does it relate to?
- What depends on it?
- What does it affect?
- What rules or constraints attach to it?
- What meaning does it carry in context?

Without taxonomy, ontology becomes unstable because the objects are not classified clearly.  
Without ontology, taxonomy becomes shallow because categories exist without relational meaning.

Fractal ontology therefore depends on a stable classification layer, and in turn makes that classification more meaningful.

## Relationship to decision-making

Fractal ontology is also important for decision-making.

A decision system does not only need options. It needs context.

For example:
- What is upstream of this decision?
- What downstream effects might it have?
- What constraints apply?
- What dependencies must be considered?
- Which actors are implicated?
- What evidence supports the decision?
- What other objects will be changed if this one changes?

Ontology helps make those questions visible.

That means ontology is not just descriptive. It is part of making a system **decision-aware** and **decision-ready**.

## Design principles

This node assumes several design principles:

### 1. Same relational logic, different scale
The same types of relationships should be able to recur across enterprise, team, workflow, document, issue, and task levels.

### 2. Semantic continuity
Meaning should not collapse when moving from one level of the system to another.

### 3. Context preservation
Objects should remain interpretable in light of their dependencies, constraints, owners, and downstream effects.

### 4. Structured extensibility
New entities and relation types should be able to enter the system without breaking the existing logic.

### 5. Human-readable and machine-readable alignment
The ontology should be interpretable by humans while still being structured enough for AI systems, machine reasoning, and governance workflows.
