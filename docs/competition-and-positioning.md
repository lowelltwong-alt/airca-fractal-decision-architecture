# AIRCA Positioning

AIRCA overlaps with several existing categories, but it should not try to be a clone of any of them.

## What AIRCA is not

AIRCA is not primarily:

- a generic workflow engine
- a BPM suite
- a pure policy engine
- only an ADR template
- only an LLM agent framework

## What AIRCA should be

AIRCA should be an **AI-native decision architecture** that combines:

- human accountability
- explicit ranking logic
- machine-readable artifacts
- GitHub-friendly records
- recursive decision structure

## Comparison points

### ADRs

ADRs capture decisions after the fact.

AIRCA should help structure the full decision cycle before and through commitment.

### Policy engines

Policy engines answer whether an action is allowed.

AIRCA should answer how a decision is framed, informed, ranked, committed, and acted on.

### Workflow engines

Workflow engines coordinate execution.

AIRCA should coordinate decision logic and durable decision memory.

### Agent frameworks

Agent frameworks coordinate model behavior.

AIRCA should define the decision grammar those systems can use.

## Best wedge into adoption

The strongest first wedge is:

**machine-readable decision artifacts for AI-assisted architecture and governance decisions**

That wedge is:

- practical
- GitHub-native
- explainable
- extensible into larger systems later
